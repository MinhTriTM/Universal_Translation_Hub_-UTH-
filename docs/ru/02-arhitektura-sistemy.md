## 1. Введение

### 1.1 Назначение

Документ описывает архитектуру программного обеспечения UTH — мультиагентной системы для автоматического перевода медиаконтента на вьетнамский язык.

### 1.2 Архитектурные цели

| Приоритет | Цель | Описание |
|-----------|------|----------|
| 1 | Модульность | Каждый агент — независимый модуль |
| 2 | Расширяемость | Лёгкое добавление новых движков и конвейеров |
| 3 | Производительность | Максимальное использование GPU |
| 4 | Надёжность | Изоляция сбоев отдельных агентов |
| 5 | Простота | Минимальные зависимости, локальная работа |

### 1.3 Заинтересованные стороны

| Роль | Интерес |
|------|---------|
| Переводчик | Быстрый и качественный перевод |
| Разработчик | Расширяемость и поддержка |
| Исследователь | Метрики и логирование |

### 2.2 Паттерн Multi-Agent

Центральный паттерн системы — **Multi-Agent Architecture** с иерархическим управлением:

```
                    ┌─────────────────┐
                    │  Director Agent  │
                    │  (Координация)   │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
        ┌─────▼─────┐ ┌─────▼─────┐ ┌─────▼─────┐
        │  Router   │ │    QA     │ │  Logger   │
        │  Agent    │ │   Agent   │ │   Agent   │
        └─────┬─────┘ └───────────┘ └───────────┘
              │
   ┌──────────┼──────────┐
   │          │          │
┌──▼───┐ ┌───▼──┐ ┌────▼───┐
│Game  │ │Manga │ │Movie   │
│Agents│ │Agents│ │Agents  │
└──────┘ └──────┘ └────────┘
```

### 2.3 Описание агентов

#### 2.3.1 Director Agent (Центральный)

| Поле | Значение |
|------|----------|
| **Роль** | Координация всех агентов |
| **Вход** | Задача пользователя |
| **Выход** | Результат перевода |
| **MiMo модель** | V2.5-Pro |

**Ответственность:**
- Приём и парсинг задач от пользователя
- Управление жизненным циклом задач
- Мониторинг состояния агентов
- Обработка ошибок и восстановление
- Формирование итогового отчёта

#### 2.3.2 Router Agent (Маршрутизатор)

| Поле | Значение |
|------|----------|
| **Роль** | Определение типа контента и маршрутизация |
| **Вход** | Входной файл/задача |
| **Выход** | Идентификатор конвейера |
| **MiMo модель** | V2.5 |

**Ответственность:**
- Анализ MIME-типа и расширения файла
- Определение игрового движка (для игр)
- Выбор оптимального конвейера
- Валидация входных данных

#### 2.3.3 Translator Agent (Переводчик)

| Поле | Значение |
|------|----------|
| **Роль** | Перевод текста на вьетнамский |
| **Вход** | Исходный текст + контекст |
| **Выход** | Переведённый текст |
| **MiMo модель** | V2.5-Pro |

**Ответственность:**
- Перевод с учётом контекста (имена, термины)
- Управление глоссарием проекта
- Пакетная оптимизация запросов
- Кэширование повторяющихся переводов

#### 2.3.4 OCR Agent

| Поле | Значение |
|------|----------|
| **Роль** | Распознавание текста на изображениях |
| **Вход** | Изображение (страница манги) |
| **Выход** | Текст + bounding boxes |
| **MiMo модель** | V2.5 (OCR-специализация) |

#### 2.3.5 Inpainting Agent

| Поле | Значение |
|------|----------|
| **Роль** | Удаление текста с изображений |
| **Вход** | Изображение + маска текста |
| **Выход** | Изображение без текста |
| **MiMo модель** | V2.5 (Inpainting-специализация) |

#### 2.3.6 Render Agent

| Поле | Значение |
|------|----------|
| **Роль** | Наложение переведённого текста |
| **Вход** | Изображение без текста + перевод + позиции |
| **Выход** | Итоговое изображение |
| **MiMo модель** | — (использует PIL/Pillow) |

#### 2.3.7 STT Agent

| Поле | Значение |
|------|----------|
| **Роль** | Распознавание речи из аудио |
| **Вход** | Аудиодорожка фильма |
| **Выход** | Текст + таймкоды |
| **MiMo модель** | V2.5 (STT-специализация) |

#### 2.3.8 TTS Agent

| Поле | Значение |
|------|----------|
| **Роль** | Синтез вьетнамской речи |
| **Вход** | Текст + таймкоды |
| **Выход** | Аудиофайл (.wav) |
| **MiMo модель** | MiMo TTS |

#### 2.3.9 Voice Agent

| Поле | Значение |
|------|----------|
| **Роль** | Управление голосами и тембрами |
| **Вход** | Профиль голоса |
| **Выход** | Настройки голоса |
| **MiMo модель** | MiMo VoiceDesign |

#### 2.3.10 VoiceClone Agent

| Поле | Значение |
|------|----------|
| **Роль** | Клонирование голоса актёра |
| **Вход** | Аудио-сэмпл (10+ сек) |
| **Выход** | Клонированный голос |
| **MiMo модель** | MiMo VoiceClone |

#### 2.3.11 QA Agent

| Поле | Значение |
|------|----------|
| **Роль** | Контроль качества |
| **Вход** | Результат перевода |
| **Выход** | Отчёт качества |
| **MiMo модель** | V2.5 |

## 3. Процессная модель

### 3.1 Конвейер «Игры» (Games Pipeline)

```
┌────────┐   ┌────────┐   ┌────────────┐   ┌────────────┐   ┌────────┐
│ Вход:  │   │ Router │   │ Game Engine│   │Translator  │   │ QA     │
│ Файл   │──►│ Agent  │──►│ Extractor  │──►│   Agent    │──►│ Agent  │──► Выход
│ игры   │   │        │   │ (12 типов) │   │ (MiMoPro)  │   │        │
└────────┘   └────────┘   └────────────┘   └────────────┘   └────────┘
                              │
                              ▼
                    ┌──────────────────┐
                    │ Форматы:         │
                    │ .rpy .rvdata2    │
                    │ .json .xml       │
                    │ .locres .po      │
                    │ .ks .tjs и др.   │
                    └──────────────────┘
```

**Этапы:**
1. **Детекция движка** — Router определяет тип движка по расширению/содержимому
2. **Извлечение текста** — парсер движка извлекает строки перевода
3. **Перевод** — Translator переводит текст с контекстом
4. **Проверка** — QA проверяет целостность и качество
5. **Сборка** — переведённые строки записываются обратно в формат движка

### 3.2 Конвейер «Манга» (Manga Pipeline)

```
┌────────┐   ┌────────┐   ┌────────┐   ┌────────────┐   ┌────────┐   ┌────────┐
│ Вход:  │   │ Router │   │  OCR   │   │ Translator │   │Inpaint │   │ Render │
│ Страница│──►│ Agent  │──►│ Agent  │──►│   Agent    │──►│ Agent  │──►│ Agent  │──►
│ манги  │   │        │   │        │   │            │   │        │   │        │
└────────┘   └────────┘   └────────┘   └────────────┘   └────────┘   └────────┘
                                │             │              │            │
                                ▼             ▼              ▼            ▼
                           Текст +      Переведённый    Изображение  Итоговое
                           координаты   текст           без текста   изображение
```

**Этапы:**
1. **OCR** — распознавание текста и определение bounding boxes
2. **Перевод** — перевод распознанного текста
3. **Инпейнтинг** — удаление оригинального текста с изображения
4. **Рендер** — наложение вьетнамского текста с сохранением стиля
5. **QA** — проверка качества изображения и текста

### 3.3 Конвейер «Фильмы» (Movies Pipeline)

```
┌────────┐   ┌────────┐   ┌────────┐   ┌────────────┐   ┌────────┐
│ Вход:  │   │ Router │   │  STT   │   │ Translator │   │  TTS   │
│ Видео  │──►│ Agent  │──►│ Agent  │──►│   Agent    │──►│ Agent  │──►
│        │   │        │   │(Whisper)│   │            │   │        │
└────────┘   └────────┘   └────────┘   └────────────┘   └────────┘
                                │             │              │
                                ▼             ▼              ▼
                          Субтитры +    Переведённые    Аудио на
                          таймкоды      субтитры       вьетнамском

                    ┌────────┐   ┌────────┐   ┌────────┐
                    │ Voice  │   │VoiceCl │   │ FFmpeg │
                    │ Agent  │──►│ Agent  │──►│ Микс   │──► Выход: Видео с дубляжом
                    │        │   │        │   │        │
                    └────────┘   └────────┘   └────────┘
```

**Этапы:**
1. **Извлечение аудио** — FFmpeg отделяет аудиодорожку
2. **STT** — распознавание речи с таймкодами
3. **Перевод** — перевод субтитров
4. **TTS** — синтез вьетнамской речи
5. **Voice Clone** — применение тембра оригинального голоса
6. **Микс** — FFmpeg объединяет новую аудиодорожку с видео

### 5.2 Базовый класс агента

```python
# agents/base.py
from abc import ABC, abstractmethod
from typing import Any, Optional
from dataclasses import dataclass, field
from enum import Enum
import asyncio
import logging

class AgentStatus(Enum):
    IDLE = "idle"
    BUSY = "busy"
    ERROR = "error"
    SHUTDOWN = "shutdown"

@dataclass
class AgentMessage:
    """Сообщение между агентами"""
    sender: str
    receiver: str
    action: str
    payload: dict
    message_id: str = ""
    timestamp: float = 0.0

@dataclass
class AgentResponse:
    """Ответ агента"""
    success: bool
    data: Any = None
    error: Optional[str] = None
    metrics: dict = field(default_factory=dict)

class BaseAgent(ABC):
    """Базовый класс для всех AI-агентов UTH"""

    def __init__(self, name: str, model_name: str):
        self.name = name
        self.model_name = model_name
        self.status = AgentStatus.IDLE
        self.logger = logging.getLogger(f"uth.agent.{name}")

    @abstractmethod
    async def process(self, message: AgentMessage) -> AgentResponse:
        """Обработка входящего сообщения"""
        pass

    @abstractmethod
    async def initialize(self) -> bool:
        """Инициализация агента (загрузка модели)"""
        pass

    async def health_check(self) -> dict:
        """Проверка состояния агента"""
        return {
            "name": self.name,
            "status": self.status.value,
            "model": self.model_name
        }

    async def shutdown(self):
        """Корректное завершение работы"""
        self.status = AgentStatus.SHUTDOWN
        self.logger.info(f"Агент {self.name} завершает работу")
```

## 6. Архитектурные решения

### 6.1 Записи архитектурных решений (ADR)

#### ADR-01: Выбор Multi-Agent Architecture

| Поле | Значение |
|------|----------|
| **Статус** | Принято |
| **Контекст** | Система должна обрабатывать три типа контента с разными требованиями |
| **Решение** | Использовать 10 специализированных агентов под управлением Director |
| **Обоснование** | Разделение ответственности, независимое масштабирование, изоляция сбоев |
| **Последствия** | Сложность координации, overhead на коммуникацию между агентами |

#### ADR-02: Выбор MiMo V2.5 как AI-платформы

| Поле | Значение |
|------|----------|
| **Статус** | Принято |
| **Контекст** | Нужна единая платформа для LLM, TTS, Voice Clone, OCR |
| **Решение** | Использовать Xiaomi MiMo V2.5 (8 моделей) |
| **Обоснование** | Единый экосистема, оптимизация для GPU, поддержка вьетнамского |
| **Последствия** | Зависимость от одного вендора, необходимость GPU NVIDIA |

#### ADR-03: Выбор SQLite вместо PostgreSQL

| Поле | Значение |
|------|----------|
| **Статус** | Принято |
| **Контекст** | Система работает локально, без сетевого доступа |
| **Решение** | Использовать SQLite |
| **Обоснование** | Нулевая настройка, файловая БД, достаточная производительность |
| **Последствия** | Ограничение на многопользовательский режим |

#### ADR-04: Выбор FastAPI

| Поле | Значение |
|------|----------|
| **Статус** | Принято |
| **Контекст** | Нужен быстрый асинхронный фреймворк с поддержкой WebSocket |
| **Решение** | FastAPI + uvicorn |
| **Обоснование** | Async/await, автоматическая документация OpenAPI, WebSocket |
| **Последствия** | Зависимость от экосистемы Python |

#### ADR-05: Паттерн Pipeline для конвейеров

| Поле | Значение |
|------|----------|
| **Статус** | Принято |
| **Контекст** | Каждый тип контента проходит через последовательность этапов |
| **Решение** | Реализовать паттерн Pipeline с возможностью параллельного выполнения |
| **Обоснование** | Чёткая последовательность, логирование каждого этапа |
| **Последствия** | Дополнительная абстракция, необходимость обработки ошибок на каждом этапе |
